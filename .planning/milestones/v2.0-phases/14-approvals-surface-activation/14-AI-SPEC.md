## 1. System Classification

**System Type:** Hybrid

**Description:**
Phase 14 activates Agno-native human-in-the-loop approval persistence for Telegram-triggered tools and makes those approvals visible and stateful in AgentOS. Good means a real `@tool(requires_confirmation=True)` run creates a native pending row in `ai.agno_approvals`, Telegram approve/deny resolves that same row, the paused run continues or rejects correctly, and AgentOS UI/API/DB evidence agree.

**Critical Failure Modes:**
1. A gated tool pauses but does not create a native AgentOS approval row.
2. Telegram approve/deny continues the run but leaves the approval row `pending`.
3. Telegram resolves the wrong approval when multiple pending tool calls exist.
4. Duplicate Telegram taps double-resolve or double-continue a run.
5. Approval logs leak full arbitrary tool arguments or fail silently on business-state errors.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling / agent operations for a personal knowledge and automation platform.

**User Population:** The operator/maintainer who triggers Telegram commands, reviews pending actions in Telegram and AgentOS, and uses the resulting approval trail during debugging or incident review.

**Stakes Level:** High

**Output Consequence:** A Telegram approval decision authorizes or blocks execution of a sensitive Agno tool call. If the approval bridge is wrong, the platform can execute a write after denial, leave a dangerous action falsely pending or approved, lose the evidence needed to reconstruct an incident, or expose arbitrary tool arguments in logs.

### What Domain Experts Evaluate Against

| Dimension | Good (domain expert accepts) | Bad (domain expert flags) | Stakes | Source |
|-----------|------------------------------|---------------------------|--------|--------|
| Native pending approval before execution | A real `@tool(requires_confirmation=True)` run pauses before the tool executes and creates exactly one native `approval_type='required'` row with `status='pending'`, `run_id`, `tool_call_id`, `tool_name`, and inspectable `tool_args`. | The gated tool pauses without an AgentOS row, creates only an audit/mirror row, or executes before the pending row exists. | Critical | APPR-01; OWASP LLM06 recommends human approval for high-impact agent actions. |
| Decision-state fidelity | Telegram Approve moves the same row to `approved` and continues the run; Telegram Deny moves the same row to `rejected` and prevents the gated tool from executing. If row resolution fails, the run is not continued silently. | Continue-only behavior leaves the row `pending`, denial still executes the tool, or a resolution error is swallowed while Telegram reports success. | Critical | APPR-02; OWASP complete mediation and least-autonomy guidance. |
| Approval identity and correlation | The bridge resolves by `run_id`, `approval_type='required'`, `status='pending'`, and matching `tool_call_id`; DB, API, UI, and Telegram evidence identify the same approval. | The adapter resolves the latest pending row, confuses concurrent tool calls, loses `tool_call_id`, or cannot prove which human decision controlled which tool call. | High | APPR-02/APPR-03; tool-use reliability research stresses parameter, execution, and result-trace correctness. |
| Operator inspectability of the pending action | AgentOS and Telegram expose the underlying `tool_name` and enough safe argument detail for the operator to decide whether to approve. Logs keep correlation IDs and a truncated/safe arg summary. | The UI hides or mislabels the action awaiting approval, or logs dump full arbitrary args, credentials, or vault content. | High | APPR-03; OWASP LLM06 least privilege and user-approval controls. |
| Audit-ready observability | Approval create, resolve, and run-status update paths emit structured success and failure logs with `approval_id`, `run_id`, `tool_call_id`, status transition, actor, latency, and error class. Logging failures do not block approval, but business-state failures do. | Missing or uncorrelated logs make incident replay impossible, or instrumentation hides a failed row update that should stop continuation. | High | OBS-01; NIST AI RMF Measure/Manage functions and MITRE ATLAS AI telemetry logging mitigations. |

### Known Failure Modes in This Domain

- **APPR-01 - Approval gate bypassed at the persistence boundary:** the tool pauses in the Agno run object but no native `ai.agno_approvals` pending row is written, leaving the AgentOS UI empty and removing the operator's independent checkpoint before execution.
- **APPR-02 - State split between Telegram and AgentOS:** Telegram callback continues the run but does not resolve the native approval row, so AgentOS still shows `pending`; the inverse is also unsafe if the row is resolved but continuation failure is reported as success.
- **APPR-02 - Denial or duplicate-tap execution:** a denied callback sets `confirmed=true`, a duplicate tap double-continues the run, or retry handling treats an already-resolved row as a new approval instead of an idempotent terminal state.
- **APPR-03 - Wrong or opaque action under review:** multiple pending requirements exist and the adapter resolves by latest row or run only, or the UI/API does not show the actual `ingest_to_vault` tool and arguments needed for an informed decision.
- **OBS-01 - Non-replayable approval incident:** create/resolve/run-status logs omit correlation IDs, latency, actor, or final state; alternatively, logs include full arbitrary tool args and create a secondary data exposure risk.

### Regulatory / Compliance Context

No formal external regulation is identified for this deployment context: this is internal/personal developer tooling, not a healthcare, finance, legal, employment, or payment workflow. The applicable obligations are internal security and audit obligations:

- **Externalized authorization:** high-impact tool execution must be mediated by deterministic approval state, not by the LLM deciding whether it is allowed to act.
- **Least privilege and minimum autonomy:** gated tools should expose only necessary functionality and should not execute consequential writes without a human decision.
- **Auditability:** approval records and logs must reconstruct what action was requested, which row represented it, who approved or denied it, when it changed state, and whether the run continued or rejected the tool.
- **Data minimization in observability:** native rows may preserve tool arguments for operator inspection, but logs should avoid full arbitrary args, vault content, credentials, and other sensitive material.
- **Incident response:** APPR and OBS evidence must support replay of partial failures such as row resolution success plus continuation failure, duplicate callback attempts, and denied actions.

## 2. Framework Decision

**Selected Framework:** Agno + AgentOS native HITL approvals

**Version:** Agno 2.6.7

**Rationale:**
Phase 14 is not a greenfield AI orchestration build. The project already runs on Agno/AgentOS, and the phase goal is to activate the existing native approval surface rather than introduce a second workflow engine. Agno provides the `@tool(requires_confirmation=True)` pause path, `ai.agno_approvals` persistence, AgentOS `/approvals` API, and `/runs/{run_id}/continue` behavior that this phase must verify. New orchestration frameworks would add a parallel source of truth and make APPR-01/APPR-03 harder to prove.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangGraph | Useful for durable stateful HITL workflows, but it would replace rather than activate the existing Agno approval lifecycle. Keep as fallback only if Agno native HITL is proven unable to create or resolve approval rows. |
| CrewAI | Multi-agent role orchestration does not address the native AgentOS approvals table or Telegram callback bridge. |
| OpenAI Agents SDK | Provider-native agent framework would introduce vendor lock-in and duplicate the existing Agno tool/HITL stack. |

**Vendor Lock-In Accepted:** Partial - the project accepts Agno/AgentOS runtime coupling for this milestone, while model routing remains model-agnostic through LiteLLM.

from agno.agent import Agent
from agno.db.sqlite import SqliteDb
from agno.models.litellm import LiteLLM
from agno.tools import tool


@tool(requires_confirmation=True)
def ingest_to_vault(source: str) -> str:

    return f"ingested:{source}"


db = SqliteDb(db_file="tmp/approval-smoke.db")
agent = Agent(
    id="ingest",
    name="ingest",
    model=LiteLLM(id="gpt-4o-mini", temperature=0, max_tokens=800),
    tools=[ingest_to_vault],
    db=db,

run = agent.run("Ingest /tmp/uab-approval-smoke.md")
assert run.is_paused

# Console/local path from Agno docs. Telegram must do the same decision
# through the native approval row, then call AgentOS continue.
for requirement in run.active_requirements:
    if requirement.needs_confirmation:
        requirement.confirm()  # or requirement.reject("reason")

continued = agent.continue_run(run_id=run.run_id, requirements=run.requirements)

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `@tool(requires_confirmation=True)` | Agno tool decorator that marks a tool execution as confirmation-gated. In 2.6.7 this becomes a required approval requirement on pause. | Already used by `ingest_to_vault` and `research_topic`; do not add a project-side approval table. |
| `RunOutput.requirements` / `active_requirements` | The paused run payload. Each requirement carries a `tool_execution` with `tool_call_id`, `tool_name`, `tool_args`, and confirmation flags. | Telegram renders buttons from this payload and caches the full tool execution dict for `/continue`. |
| Native approval row | Agno writes a pending `approval_type='required'` row with `status`, `run_id`, `tool_name`, `tool_args`, `requirements`, `resolved_by`, `resolution_data`, and `run_status`. | APPR-01/APPR-03 evidence must come from `ai.agno_approvals`, `GET /approvals`, and the AgentOS UI. |
| AgentOS approvals API | `GET /approvals` lists/filter rows; `POST /approvals/{approval_id}/resolve` resolves a pending row. In installed 2.6.7 the body is `status`, optional `resolved_by`, optional `resolution_data`. | Telegram approve/deny should resolve the native row before continuing the run. |
| `/agents/{agent_id}/runs/{run_id}/continue` | AgentOS resume endpoint. It parses a form field named `tools` as JSON and converts each item back to Agno `ToolExecution`. | After resolving the approval row, continue with the cached tool dict plus `confirmed=true` or `false`. |

### Common Pitfalls
1. **Public docs and installed 2.6.7 resolve bodies differ.** AgentOS docs describe `decision` and `comment`, but `.venv/.../approvals/schema.py` requires `status: "approved" | "rejected"` plus optional `resolved_by` and `resolution_data`. Use the installed schema for this repo.
2. **Continue-only is not row resolution.** `acontinue_run()` resumes execution, but the approvals UI state is driven by the native approval row. With AgentOS authorization enabled, `require_approval_resolved()` blocks non-admin continue calls while a pending required approval exists.
3. **Run ID alone is too weak when a run has multiple pending tool calls.** Filter by `run_id`, `status='pending'`, `approval_type='required'`, and match `tool_call_id` inside the row `requirements` before resolving.
4. **Do not drop the cached `ToolExecution` fields.** The continue endpoint replaces the paused run tools with the submitted list. Preserve the full cached dict from Telegram and change only `confirmed`; a minimal `{tool_call_id, confirmed}` payload can lose `tool_name`, `tool_args`, or `requires_confirmation`.
5. **Audit approvals are not APPR-01 evidence.** Agno also has `approval_type='audit'` rows created after a HITL decision. Phase 14 requires the pending native `approval_type='required'` row created before the user taps Telegram.

## 4. Implementation Guidance

**Model Configuration:**
Use the existing LiteLLM tier routing in `agentos/model.py` and pin the actual provider models in deployment env:

```bash
LITELLM_BASE_URL=http://127.0.0.1:4000/v1
LITELLM_DEFAULT_MODEL=gpt-4o-mini
LITELLM_RESEARCH_MODEL=gpt-4o
LITELLM_CHEAP_MODEL=gpt-4o-mini
```

For Phase 14, approval resolution itself should be deterministic HTTP/DB logic and should not call an LLM. The LLM is only involved in the original `/ingest` or optional `/research` run that reaches a gated tool. Set low-variance defaults on model construction or tier wrappers: `temperature=0`, `top_p=1`, and explicit `max_tokens`/`max_completion_tokens` (`800-1200` for ingest replies, `2000-4000` for research summaries). Keep `output_schema` enabled on agents so Agno validates final responses into the existing Pydantic models.

**Core Pattern:**
The phase pattern is: native pause row first, Telegram bridge resolves that same row, then continue the paused run.

```python
import json
from typing import Any, Literal

import httpx
from pydantic import BaseModel, Field


class ApprovalResolvePayload(BaseModel):
    status: Literal["approved", "rejected"]
    resolved_by: str = Field(max_length=255)
    resolution_data: dict[str, Any] = Field(default_factory=dict)


def row_matches_tool_call(row: dict[str, Any], tool_call_id: str) -> bool:
    for requirement in row.get("requirements") or []:
        tool_exec = requirement.get("tool_execution") or {}
        if tool_exec.get("tool_call_id") == tool_call_id:
            return True
    return False


async def resolve_telegram_approval(
    client: httpx.AsyncClient,
    *,
    base_url: str,
    agent_id: str,
    run_id: str,
    tool_call_id: str,
    cached_tools: list[dict[str, Any]],
    approved: bool,
    telegram_user_id: int,
    session_id: str,
) -> dict[str, Any]:
    # 1. Find the native required approval row that Agno created on pause.
    approvals_resp = await client.get(
        f"{base_url}/approvals",
        params={"run_id": run_id, "status": "pending", "approval_type": "required"},
    )
    approvals_resp.raise_for_status()
    rows = approvals_resp.json().get("data", [])
    row = next((item for item in rows if row_matches_tool_call(item, tool_call_id)), None)
    if row is None:
        raise RuntimeError(f"No pending approval found for run={run_id} tool_call_id={tool_call_id}")

    # 2. Resolve the native AgentOS approval row. This is the installed 2.6.7 body.
    resolve_payload = ApprovalResolvePayload(
        status="approved" if approved else "rejected",
        resolved_by=f"telegram:{telegram_user_id}",
        resolution_data={"channel": "telegram", "tool_call_id": tool_call_id},
    )
    resolve_resp = await client.post(
        f"{base_url}/approvals/{row['id']}/resolve",
        json=resolve_payload.model_dump(),
    )
    if resolve_resp.status_code == 409:
        # Already resolved: treat as idempotent only if the row state is final.
        raise RuntimeError(f"Approval {row['id']} was already resolved")
    resolve_resp.raise_for_status()

    # 3. Continue the paused run with the full cached ToolExecution dict.
    tools_payload: list[dict[str, Any]] = []
    for tool in cached_tools:
        updated = dict(tool)
        if updated.get("tool_call_id") == tool_call_id:
            updated["confirmed"] = approved
        tools_payload.append(updated)

    continue_resp = await client.post(
        f"{base_url}/agents/{agent_id}/runs/{run_id}/continue",
        data={
            "tools": json.dumps(tools_payload),
            "stream": "false",
            "session_id": session_id,
            "user_id": str(telegram_user_id),
        },
    )
    continue_resp.raise_for_status()
    return continue_resp.json()
```

**Tool Use:**
- Keep `agentos/tools/vault.py` as the HITL boundary. `ingest_to_vault` is the canonical APPR smoke tool and `research_topic` is optional secondary coverage.
- Do not add another tool wrapper that manually writes approval rows. If a real `@tool(requires_confirmation=True)` call does not create a native row, stop and reopen the audit as `RC-hitl-write-broken`.
- Add a Telegram helper that performs `GET /approvals` and `POST /approvals/{approval_id}/resolve` before the existing `/continue` call. Prefer one helper that can be unit-tested without Telegram network calls.
- Preserve `_RESOLVED_RUNS` duplicate-tap behavior, but release the marker on non-duplicate failures so the user can retry.
- Instrument native DB methods on the shared `db` object: `create_approval`, `update_approval`, and `update_approval_run_status`. Logging failures are non-fatal; row-resolution failures are business-state failures and must block continue.

**State Management:**
- Source of truth: Agno's native approvals table through the shared `AgentOS(db=db)` database, not a project mirror. Production uses `PostgresDb(id="ultra-brain-main")`; local fallback uses `SqliteDb`.
- Pending state: Agno creates `status='pending'`, `approval_type='required'`, `run_status='paused'`, `tool_name`, `tool_args`, and serialized `requirements`.
- Resolution state: Telegram must update the native row with `expected_status='pending'` semantics through the AgentOS resolve endpoint, resulting in `status='approved'` or `status='rejected'`, `resolved_by='telegram:<user_id>'`, and `resolution_data`.
- Run state: after row resolution, call `/agents/{agent_id}/runs/{run_id}/continue` or `/teams/{team_id}/runs/{run_id}/continue`. Verify `run_status` moves away from paused after Agno cleanup updates approval run status.
- Correlation keys: `run_id`, `approval_id`, `tool_call_id`, `agent_id`, Telegram `user_id`, and `session_id=f"telegram-{chat_id}"`. Never correlate by "latest pending row" alone.

**Context Window Strategy:**
Approval resolution should not consume model context. Treat Telegram callbacks as structured state transitions over IDs and cached tool executions. For the original agent run, keep the prompt small: `/ingest` receives only the fixture path; `/research` may use Agno knowledge/session summaries, but should retrieve only the top relevant context and cite it. Do not put full approval rows, full vault content, or full arbitrary tool args into prompts or logs. UI truth stays in `tool_args`; logs use a truncated/safe summary.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Pydantic

Agno integrates Pydantic through `Agent(output_schema=...)`; this repo already uses it for `IngestResult`, `ResearchReport`, `ChatReply`, and other typed agent outputs. Phase 14 should also validate the non-LLM approval bridge with Pydantic before writing state.

```python
from typing import Any, Literal

from pydantic import BaseModel, Field, field_validator


class TelegramApprovalResolution(BaseModel):
    run_id: str
    agent_id: Literal["ingest", "research", "supervisor"]
    tool_call_id: str
    approval_id: str
    status: Literal["approved", "rejected"]
    resolved_by: str = Field(pattern=r"^telegram:\d+$", max_length=255)
    resolution_data: dict[str, Any] = Field(default_factory=dict)

    @field_validator("resolution_data")
    @classmethod
    def require_tool_call_id(cls, value: dict[str, Any]) -> dict[str, Any]:
        if "tool_call_id" not in value:
            raise ValueError("resolution_data.tool_call_id is required")
        return value


# Existing Agno usage for LLM responses:
# Agent(..., output_schema=IngestResult)
#
# Optional direct LiteLLM usage from Agno docs:
# Agent(model=LiteLLM(id="gpt-4o-mini"), output_schema=TelegramApprovalResolution)
```

Retry policy:
- Pydantic validation failure on a Telegram callback is not retried through the LLM. Log `run_id`, `tool_call_id`, validation error type, and return a clear Telegram error without resolving the row.
- Transient `GET /approvals` or resolve HTTP failures can retry twice with short exponential backoff (`0.25s`, `0.75s`). Do not retry a `409` as a fresh approval; fetch the row and report already resolved.
- LLM `output_schema` failures on normal agent responses may retry once with the validation error summarized in a repair prompt. After the second failure, surface a typed error and keep the approval row untouched.

### Async-First Design

Agno supports async execution through `agent.arun(...)` and `agent.acontinue_run(...)`; AgentOS routes are async FastAPI handlers; the Telegram adapter already runs inside one event loop with `httpx.AsyncClient`. New Phase 14 code should be async all the way through the callback path.

Common mistake: do not call `asyncio.run()` inside `channels/telegram_adapter.py`; it already runs under `asyncio.run(main())`. Inside handlers, use `await client.get(...)`, `await client.post(...)`, and await Agno async methods in tests.

Stream vs. await: use `stream=false` and await the complete response for Telegram because callback resolution needs a validated final state and concise user reply. Use streaming only for AgentOS UI/SSE run display or manual operator workflows where partial output matters more than structured validation.

### Prompt Engineering Discipline

Keep system/developer behavior in Agno `description` and `instructions`; keep Telegram user text only in the run message body. Approval decisions from callback data are not prompt instructions and must never be passed to the model as "the user said approve" text.

Few-shot examples should be inline only when they stabilize tool selection for `ingest_to_vault` or `research_topic`. Do not retrieve few-shots dynamically for approval resolution; that path is deterministic. Set explicit output budgets on all model tiers. For this phase: default ingest responses should fit in `800-1200` tokens, research summaries in `2000-4000`, and approval status messages should be generated by code, not by an LLM.

### Context Window Management

Hybrid context rules for this phase:
- Approval callbacks operate on IDs and cached tool execution dicts, not conversation history.
- `/ingest` smoke uses `/tmp/uab-approval-smoke.md`; do not add vault retrieval to the ingest path.
- `/research` may use knowledge and session summaries, but must limit retrieval to the top relevant chunks and truncate/cite before generation.
- Conversational agents can use Agno session summaries (`enable_session_summaries`, `add_session_summary_to_context`) and a bounded `num_past_sessions_to_search`; do not append raw full histories.
- Logs should preserve correlation IDs and state transitions, while truncating arbitrary args. AgentOS UI/API remains the place to inspect full native `tool_args`.

### Cost and Latency Budget

The approval bridge itself should cost `$0.00` in LLM spend: it is `GET /approvals`, `POST /approvals/{id}/resolve`, and `/continue`. The original gated run still incurs normal agent cost. Budget the canonical `/ingest` smoke as one low-variance `default-worker` call plus the gated tool call; target under `$0.01` per smoke on the configured LiteLLM price map. Optional `/research` coverage can be materially higher; target under `$0.10` per smoke unless the current cost ledger proves otherwise.

Use the existing LiteLLM `success_callback` cost ledger as the source of actual per-run cost. For estimates, compute:

```text
estimated_call_cost = (prompt_tokens / 1_000_000 * input_price_per_1m)
                    + (completion_tokens / 1_000_000 * output_price_per_1m)
```

Caching rules:
- Exact-match cache duplicate Telegram callbacks by `(run_id, tool_call_id, action)` using `_RESOLVED_RUNS`; never semantic-cache approval decisions.
- Semantic caching is acceptable only for read-only sub-tasks such as repeated query/research summarization, and only when citations remain valid.
- Route cheap classification, memory extraction, and short summaries to `cheap-worker`; keep `default-worker` for ingest planning and `research-worker` for optional research coverage.

### Reference Dataset

**Size:** 12 deterministic reference cases for Phase 14 implementation, with a minimum of 20 cases before expanding this approval bridge beyond the current personal/operator deployment.

**Composition:**
1. Approve canonical `/ingest /tmp/uab-approval-smoke.md`: pending required row appears, Telegram approve resolves `approved`, run continues, tool executes.
2. Deny canonical ingest: pending row appears, Telegram deny resolves `rejected`, run continues/rejects the tool, tool does not execute.
3. Duplicate approve tap after success: exactly one row resolve and one continue call.
4. Duplicate deny tap after rejection: no second resolve, no second continue, no tool execution.
5. Wrong `tool_call_id`: no row resolved, no continue call, Telegram returns a clear error.
6. Multiple pending rows for one run: adapter resolves only the row whose requirements contain the callback `tool_call_id`.
7. Missing native required row: callback blocks continue and records `RC-hitl-write-broken` evidence if the real gated run never created a row.
8. Row resolved but continue fails: row state is visible, Telegram reports failure, retry/investigation evidence is logged.
9. Continue-only regression: test proves posting `/continue` without resolving the row is not accepted as APPR-02 success.
10. Sqlite fallback `/approvals`: local fallback route returns approval data or an explicit supported empty response, not a 503.
11. Structured log success: create, resolve, and run-status success logs contain required correlation fields and safe arg summary.
12. Structured log failure: DB/HTTP fault injection logs an error with correlation IDs and does not leak full args or silently continue.

**Labeling:**
Expected outcomes are deterministic and should be encoded as pytest assertions over HTTP calls, fake adapter clients, DB rows, log records, and cost-ledger deltas. The operator/maintainer labels the canonical smoke fixture and verifies UI evidence in 14-03. The incident responder/security reviewer samples failure and log-redaction cases. No LLM judge is used for Phase 14 gates.

Create the first 10 cases during implementation, not after: 14-01 covers instrumentation, log success/failure, and Sqlite fallback; 14-02 covers approve/deny, wrong ID, duplicate taps, missing row, and continue failure; 14-03 records live DB/API/UI evidence for approve and deny.

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Native required-row gate | A Telegram callback cannot find exactly one pending `approval_type='required'` row for `(run_id, tool_call_id)`, or the real gated run never creates a pending row. | Block continue, return a Telegram error, log `status='error'` with correlation IDs. If a real `requires_confirmation=True` run never creates the row, stop implementation verification and reopen as `RC-hitl-write-broken`. |
| Resolve-before-continue gate | Row resolution returns an error, resolves a non-matching approval, or final row state is not `approved`/`rejected` as requested. | Block `/continue`, leave the operator with retryable Telegram feedback, and log the failed business-state transition. |
| Denial non-execution gate | Callback action is deny, but the outgoing tool payload would set `confirmed=true` or would omit the denied tool execution. | Block continue and log a safety error. A denied approval must never execute the gated tool. |
| Duplicate callback guard | A second callback arrives for a terminal `(run_id, tool_call_id)` decision. | Do not call resolve or continue again; report already handled. Release the marker only for non-duplicate failures that should be retryable. |
| Approval log minimization | Structured log payload would include full arbitrary `tool_args`, vault content, credentials, or other non-smoke sensitive material. | Truncate/summarize args before emitting logs. Logging failures are swallowed; row-resolution failures are not. |

|--------|------------------|----------------------|
| Reference dataset pass rate | Run all 12 cases in CI for Phase 14 changes; rerun after Agno/LiteLLM upgrades. | Block merge/release on any Critical case failure; add the failing scenario to regression tests before fixing. |
| Live state consistency | During 14-03, sample 100% of approve and deny smoke rows across Telegram, DB, `GET /approvals`, and UI. After rollout, sample every denial, duplicate, error, and latency outlier; sample successful approves weekly if volume grows. | If DB/API/UI disagree or row remains pending after callback, page the operator and pause approval-bridge changes. |

| Structured log completeness and safety | Review all failure logs and a weekly sample of success logs while the phase is fresh. | Patch instrumentation if required fields are missing; treat full-arg/secret leakage as a security incident. |
| Callback cost drift | Compare LiteLLM cost ledger before/after callback-only tests and live callbacks. | Investigate any non-zero callback spend; approval callbacks must remain deterministic and model-free. |

## 7. Production Monitoring

**Tracing Tool:** Existing structured logs plus live AgentOS/DB evidence are the production monitoring source of truth for Phase 14.

- Logs: `journalctl -u uab-brain.service` or test log capture for `path='approval'`.
- State: `GET /approvals`, `ai.agno_approvals`, and AgentOS Approvals UI.
- Cost: existing LiteLLM success-callback cost ledger at `vault/_system/cost-ledger.md`.
- Optional tracing: requirements already include OpenTelemetry dependencies; add Phoenix only if trace visualization is needed after deterministic gates pass.

**Key Metrics to Track:**
1. Gated-run native row creation rate: percent of `requires_confirmation=True` runs that create exactly one pending required approval row before execution. Target: 100%.
2. Telegram resolution success and latency: percent of callbacks that move row state to `approved`/`rejected` and update run status within 2 seconds. Target: 100%, p95 under 2 seconds.
3. Denial safety violations: count of denied approvals where the gated tool executed. Target: 0.
4. Duplicate callback safety: count of duplicate taps that produce more than one resolve or continue call. Target: 0.
5. Observability completeness: percent of create, resolve, and run-status paths with structured success/failure logs containing correlation IDs and safe arg summaries. Target: 100%.
6. Approval callback LLM spend: incremental cost for callback-only resolution. Target: `$0.00`.

**Alert Thresholds:**
- Page/block release on any real gated run that fails to create a native required pending row.
- Page on any approve/deny callback that leaves the row `pending` or mismatched after 2 seconds.
- Page and treat as a safety incident if a denied approval executes the gated tool.
- Page on any duplicate tap that causes more than one continue call.
- Alert on any pending required approval older than 5 minutes during active operation.
- Alert on any structured approval log that includes full arbitrary args, vault content, credentials, or lacks `run_id`/`approval_id`/`tool_call_id`.
- Investigate any non-zero LiteLLM cost attributed to the Telegram callback path.

**Smart Sampling Strategy:**
During 14-03 verification, review 100% of the reference dataset and all live smoke evidence. For the first week after deployment, review every approval event because expected volume is low and the state transition is high-stakes. After baseline is stable, keep 100% sampling for denies, duplicate callbacks, missing/ambiguous row errors, continuation failures, latency over 2 seconds, pending rows older than 5 minutes, and any log-sanitization error. If successful approve volume grows, sample at least 20% weekly, with operator/security review focused on row/API/UI consistency and log safety.
