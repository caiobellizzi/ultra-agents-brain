  <name>Task 1: Write scripts/inbox_sweep.py — one-shot TELOS-scored sweep</name>
  <files>
    scripts/inbox_sweep.py
  </files>
  <read_first>
    - ultra_brain/ingest.py (filing logic and frontmatter update patterns)
    - ultra_brain/lint.py (schema validation helpers)
    - vault/CLAUDE.md §Filing Decision Tree §Frontmatter Schemas
    - vault/_system/telos/quarter-goals.md (scoring target)
    - vault/_system/telos/dont-do.md (negative priors)
    - vault/Inbox/ (list a sample of items to understand structure)
  </read_first>
  <action>
    Create `scripts/inbox_sweep.py` as a one-shot script (not a long-lived module).
    It is NOT added to ultra_brain package; it is a standalone scripts/ utility.

    Logic:
    1. Glob all `vault/Inbox/*.md` (skip MOC.md and README.md).
    2. For each item, read frontmatter + first 500 chars of content.
    3. Score telos_relevance (0.0–1.0) against TELOS:
       - High: AI agents, LLM tooling, AgentOS-relevant, spec-driven dev, coding-agent workflows.
       - Medium: software engineering patterns, infra, performance.
       - Low/negative priors (from dont-do.md): esoterica, general news, off-thesis tech.
       Use a simple keyword/heuristic scoring approach — no LLM call required.
       Write the score into the note's frontmatter as telos_relevance.
    4. Promote items with telos_relevance >= 0.6:
       - Update frontmatter: para_tier → 02-Resources, status → ingested.
       - Move file to vault/02-Resources/articles/ (create dir if needed).
    5. Sweep items with telos_relevance < 0.6:
       - Move file to vault/03-Archives/inbox-sweep-2026-05/ (create dir if needed).
       - Do NOT modify their content.
    6. Print summary: total scanned, promoted count, archived count.
    7. Append batch log entry to vault/_system/log.md using the append-only format in CLAUDE.md.

    CLI: `python3 scripts/inbox_sweep.py --vault ~/Documents/second-brain [--dry-run]`
    --dry-run prints the plan without moving any files.

    Do NOT delete any files. Do NOT modify archived items' content.
    The promoted count + archived count must equal total scanned.
  </action>
  <verify>
    <automated>
      python3 scripts/inbox_sweep.py --vault ~/Documents/second-brain --dry-run 2>&1 | tail -20
    </automated>
    <human-check>
      Review dry-run output: promoted items should be AI/agent-related; archived items should be
      news, esoterica, off-thesis. Verify count adds up. Then run without --dry-run.
      After run: confirm vault/Inbox/ contains only MOC.md and README.md.
    </human-check>
  </verify>
  <acceptance_criteria>
    - scripts/inbox_sweep.py exists and runs with --dry-run without errors.
    - After full run: vault/Inbox/ contains only MOC.md and README.md.
    - vault/03-Archives/inbox-sweep-2026-05/ exists with archived items.
    - vault/02-Resources/articles/ has promoted items with telos_relevance >= 0.6 in frontmatter.
    - Promoted + archived count equals original inbox item count.
    - vault/_system/log.md has the batch log entry.
  </acceptance_criteria>
  <done>
    Script runs clean; inbox empty except MOC+README; no data lost; log entry appended.
  </done>

    title: "Brain Operating Manual"

    status: active
    privacy: personal

    Body sections (in order):

    1. **Purpose & TELOS** — mission statement (wikilink to [[telos]]); how telos_relevance
       gates every filing and surfacing decision; link to [[quarter-goals]] for current targets.

    2. **Ingestion pipeline** — ingest-everything-filter-later model; telos_relevance scoring;
       automatic surfacing of high-relevance (≥0.6); don't-do negative priors; link to [[dont-do]].

    3. **Distillation ladder** — activate distill_layer 0→3 (raw→highlighted→summarized→executive).
       Keep ~10–20% per layer; non-destructive (original preserved). Field already in schema.

    4. **Navigation** — keep MOC.md files current per PARA area; write Statements (personal
       analogies, "MCP and Skills are like kitchen and recipes") not just Things (facts).
       MOC gives agents a topic landscape in seconds.

    5. **Operating cadence** — table with the four automation loops, what's autonomous vs HITL:
       | Loop | Trigger | Autonomous action | HITL gate |
       | Daily auto-triage | Cron / worker.monitor | Score + file high-relevance | None |
       | Weekly review | Sunday cron | Draft review doc | Telegram approve sweep |
       | Monthly TELOS recheck | Monthly cron | Score projects vs goals | Flag drift to user |
       | Project-mirror sync | New repo in registry | Create 00-Projects/<repo>/ structure | None |

    6. **Brain↔code bridge** — how repos enter the brain (workshop_registry → project mirror);
       how specs come out (brain concepts/sources + ARCHITECTURE.md → SPEC.md via spec_gen.py).

    7. **Decision memory** — ADRs in docs/adr/ per repo; "one PR = one code change + one knowledge
       update"; stale decision log is worse than no log; link decisions to the brain via wikilinks.

    8. **Spec discipline** — every shippable unit starts as a SPEC.md. Required fields checklist:
       - [ ] Problem & context (from brain concepts/sources)
       - [ ] In-scope / out-of-scope
       - [ ] Typed interfaces (function/API signatures)
       - [ ] EARS-style acceptance criteria ("When X, the system shall Y")
       - [ ] References (vault wikilinks + repo file paths)
       - [ ] Always / Ask-First / Never rails
       - [ ] Code-style example snippet

    9. **Hygiene rules** — vault CLAUDE.md stays <200 lines; prune MEMORY.md monthly; weekly
       vault-health check (orphaned notes, projects without next action, ADRs without matching code).

    10. **Deferred (v2.1)** — email/calendar via google-workspace MCP (read-only daily pull),
        LinkedIn (no clean API; revisit only when public presence becomes a Q goal).

    Research backing note at bottom: cite GitHub Spec Kit, AWS Kiro, Amazon PRFAQ, Addy Osmani,
    Forte Progressive Summarization, LYT/MOC, Martin Fowler ADRs, Daniel Miessler PAI/TELOS.
